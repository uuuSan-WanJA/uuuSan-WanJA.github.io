## 처음 그린 그림

Monitor 프로젝트를 만들기 시작했을 때 머릿속에 있던 그림은 단순했다.
여러 AI 에이전트 프로젝트들이 Monitor를 통해 연결되고, Monitor가 각 프로젝트에 지시를 내리면 유기적으로 움직이는 구조. 개별 프로젝트들은 독립적으로 돌아가되, 서로가 연결된 하나의 시스템.

가능할 것 같았다.

&nbsp;

## API를 하나씩 쌓으면서

그런데 막상 Monitor에서 다른 프로젝트에 새로운 일을 시키려 할 때마다, 매번 같은 절차를 밟아야 했다. 새 기능이 필요하면 새 API 엔드포인트를 만들고, 그걸 Monitor 쪽에서 호출하도록 연결하고, 테스트하고.

작업마다 이 과정이 반복됐다. 프로젝트 간 연결이 늘어날수록, 유지해야 할 API 계층도 함께 불어났다. 계획했던 것들이 뒤로 밀리기 시작했고, 왜 이렇게 오래 걸리는지 한동안 그 이유를 정확히 짚지 못했다.

&nbsp;

## 스스로 거세한 것

뒤늦게 깨달았다. 내가 뭘 만들고 있었는지를.

LLM 기반 AI 에이전트가 기존 소프트웨어 컴포넌트와 다른 가장 큰 이유는 하나다. 정해진 함수 시그니처 없이, 대화와 맥락만으로 소통할 수 있다는 것. "이 파일의 이 부분을 이렇게 바꿔줘"가 아니라, "지금 상황이 이렇고, 너는 어떻게 생각해?" 로 시작해도 된다는 것.

나는 그 장점을 처음부터 API 레이어로 틀어막았다. 에이전트에게 말을 거는 대신, 호출할 수 있는 함수를 만들어왔다. 매번 새 기능이 생길 때마다 새 함수가 필요했던 건 당연한 일이었다. 나는 LLM을 API 서버처럼 다루고 있었다.

&nbsp;

## 다시 그리는 그림

재설계의 방향은 이 인식에서 출발했다. 주류 멀티에이전트 프레임워크는 별도 프로세스끼리 API로 통신하지 않는다. 단일 오케스트레이터가 안에서 역할을 분배하고, 대화와 프롬프트로 서브에이전트들과 소통한다.

지금까지 "프로젝트"라는 이름으로 분리해뒀던 것들을 단일 세션 오케스트레이터 안의 서브에이전트로 이식한다. Monitor가 API로 부르는 대신, Orchestrator가 Agent 툴로 호출한다. 고정된 함수 대신 컨텍스트와 지시로 움직이는 구조로.

&nbsp;

## 아까운 시간, 아깝지 않은 인사이트

솔직히 아깝다. Monitor에 쏟은 시간과, 설계가 틀렸다는 걸 깨닫는 데까지 걸린 시간 모두.

그런데 이상하게도, 잃은 것보다 얻은 게 더 크게 느껴진다. 멀티에이전트 설계에서 API 기반 연결이 왜 안 통하는지, 단일 오케스트레이터 패턴이 왜 표준이 됐는지, 이제는 몸으로 안다. 누가 설명해줬다면 그냥 넘어갔을 이야기들이, 직접 막혀보고 나니 구조로 박혔다.

새로운 길을 찾았다고 생각했는데 결국 큰 우회로였다. 그래도 우회로를 걸어본 사람만 아는 것들이 있다.

오늘 하루 동안 치명적인 실패를 두 번이나 겪었다. 합산 피해는 시간 4시간에 비용 50달러. 공통 원인은 하나, — **"이 정도는 알아서 하겠지."** — 였다.

&nbsp;

## 핵심 동작 원칙이 깨진 순간

GameMakerEngine은 GameMaker를 만들고 발전시키는 메타 시스템이다. GameMaker가 어느 정도 구현됐는지 확인하고 싶어서 테스트용 게임 제작을 지시했다. 당연히 엔진이 GameMaker에게 게임을 만들라고 시킬 줄 알았다.

결과물을 열어보니 엉망이었다. fog 설정이 빠져서 화면이 새까맣고, 한 시간 넘게 스무고개를 해서야 겨우 구동 화면을 볼 수 있었다. 그런데 진짜 문제는 퀄리티가 아니었다. 상황을 확인해 보니 **엔진이 GameMaker를 거치지 않고 직접 게임을 만들고 있었다.** 도구를 만드는 시스템이 도구를 건너뛰고 결과물을 직접 생산한 것이다.

원인은 CLAUDE.md의 구조에 있었다. "GameMaker를 통해서만 게임을 제작할 것"이라는 지시사항이 존재했지만, 컨텍스트가 길어지면서 우선순위에서 밀렸다. 핵심 규칙은 Critical로 분류해서 눈에 띄게 배치해야 한다는 교훈을 얻었고, 해당 항목은 즉시 조정했다. 나머지 지시사항의 중요도 구분은 경험을 쌓으며 점진적으로 개선할 예정이다.

### Before / After — CLAUDE.md 우회 금지 원칙

이 사건의 수습은 커밋 하나로 찍혔다. `2d28d37` — 07:41.

**Before:** "GameMaker를 통해서만 게임을 제작할 것"이라는 문장이 `작업 규칙` 섹션 어딘가에 묻혀 있었다. 다른 규칙들과 같은 우선순위, 같은 서식. 컨텍스트가 길어지면 모델이 이 줄을 건너뛸 확률은 올라간다.

**After:** 독립 섹션 `시스템 우회 금지 원칙 (CRITICAL)`이 생겼다.

```
- 게임 제작/에셋 생성/코드 생성은 반드시 GameMaker 파이프라인을 통해 실행한다.
  Claude가 직접 게임 코드, 에셋, 설계 문서를 작성하는 것은 절대 금지.
- GameMaker가 실패하면 → 실패를 보고하고 GameMaker 시스템 자체를 개선한다.
  절대 Claude가 대신 만들지 않는다.
```

핵심은 두 가지다. 첫째, **CRITICAL 태그로 우선순위를 명시**했다. 둘째, 단순 금지가 아니라 **실패 시 어떻게 해야 하는지까지 경로를 지정**했다. "하지 마"만으로는 부족하고, "대신 이렇게 해"까지 써줘야 모델이 따른다.

## 테스트 하나 실패에 전체를 처음부터

첫 번째 실패를 수습한 직후, GameMakerEngine에 GameMaker 전체 테스트를 맡겼다. 그런데 테스트가 2시간 40분이 지나도 끝나지 않았다. 그제서야 토큰 사용량을 확인했고, 그 소모량이 MAX 20x 플랜의 5시간 한도 중 90%를 넘어가고 있다는 것을 비로소 깨달았다.

문제를 확인해 보니, **30분이 넘게 걸리는 무거운 전체 테스트가, 항목 중 하나만 실패해도 처음부터 재실행되고 있었다.** 결국 한도를 초과했고, API 토큰 50달러를 추가로 구매해야 했다.

실패한 테스트만 재실행하는 건 기본적인 설계다. 엔진이 스스로 그 정도는 고려할 줄 알았지만, 현실은 달랐다.

### Before / After — 증분 실행과 테스트 절차

수습 커밋은 두 repo에 걸쳐 세 개가 찍혔다.

**GameMaker `128f3dc` — 14:10:** `증분 실행 — 성공 태스크 skip, 실패만 재실행`. Director에 task-manifest 시스템이 추가됐다. 각 태스크의 성공/실패 상태를 파일로 기록하고, 다음 실행 시 성공한 태스크는 건너뛴다. 변경량 +386줄, 테스트 200줄 포함.

**Engine `3dfc063` — 14:28:** `세션/테스트 절차 최적화 — 토큰 절약을 위한 단계별 테스트 실행 규칙`. CLAUDE.md에 상황별 테스트 범위가 명문화됐다.

```
- 세션 시작: git pull + git log만. 전체 테스트는 돌리지 않음.
- 작업 중: 변경한 파일 관련 테스트만 실행.
- 커밋 직전: Engine vitest run (~3초). GameMaker UE빌드 제외 (~10초).
- push 전: GameMaker 전체 테스트는 push 직전 1회만.
- 파이프라인 실행: 사용자 명시적 요청 시에만, 최소 범위로.
```

**Engine `a45d6eb` — 14:36:** `LLM 토큰 최적화 — 페르소나 3개 필터 + 프롬프트 압축 + 모델 haiku 전환`. 비용 자체를 줄이는 구조 변경까지 함께 들어갔다.

세 커밋이 26분 사이에 연속으로 찍혔다. 문제를 인지한 후 코드 수정(증분 실행) → 규칙 수정(테스트 절차) → 비용 구조 변경(토큰 최적화)이 한 호흡에 이루어진 것이다.

## "알아서 하겠지"를 시스템으로 바꾸기

두 실패의 교훈은 명확하다. AI 에이전트에 대한 **암묵적 기대는 위험하다.** "이 정도는 당연히"라는 생각이 들 때가 가장 경계해야 할 순간이다.

그렇다고 모든 과정을 사람이 직접 검토하는 방식으로 돌아갈 생각은 없다. 그건 AI 에이전트를 쓰는 의미를 포기하는 것이다. 답은 **검증 자체를 자동화하는 것** — 사람이 감시하는 게 아니라, 검증 프로세스를 시스템에 내장해서 AI가 AI를 점검하게 만드는 것이다.

"알아서 하겠지"를 "알아서 검증하게 만들자"로 바꾸는 것. 그게 오늘 50달러로 산 교훈이다.
